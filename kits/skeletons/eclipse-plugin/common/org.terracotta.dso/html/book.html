Terracotta Distributed Shared Objects (DSO)

With Terracotta DSO, you can craft distributed applications whose data is transparently shared, or mirrored, across many Java virtual-machines. This transparency means, among other things, that you do not write any code. You simply configure your application in a declarative way, specifying which fields are to be shared and which methods are to act as distributed semaphores during modification of that data. Terracotta DSO handles the task of distributing your changes to other like-configured clients running in different virtual machines, possibly on different hosts.

The benefits of the DSO architecture includes, but is not limited to, a natural, standard development process without the burdens of yet another application programming interface with exceptions that must be thrown and caught.

Using DSO

Accomplishing transparent data sharing with DSO is similar to creating a normal application meant to run in a single Java virtual machine. You write your classes as you always have and only later, at your convenience, do you target your application at the Terracotta runtime.

The Terracotta DSO Eclipse Plugin serves to make that transition as straightforward as possible. The plugin provides several different ways to convey your particular configuration needs, from forms&fields editors to direct manipulation of the elements in your Java code editor.

Having expressed how you want your code to integrate with DSO, you can fire up the Terracotta Server, run multiple instances of your application, and see how DSO mirrors your data such that it appears as a single copy to your application. Your application has no knowledge that the data it holds is being managed under the hood by a combination of DSO classes in your client and the Terracotta Server.
